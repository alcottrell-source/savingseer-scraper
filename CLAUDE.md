# Savingseer / Tide — Claude context

## Project overview
Single-file static web app (`index.html`) deployed to Vercel. No build step. Talks directly to Supabase for auth and data. Pipeline lives in separate Node scripts: `score.js` (compute Tide Score per centre) → `summarise.js` (Claude-written 1–2 sentence Centre Intelligence narrative). **Sale state is admin-verified only** — an operator confirms each brand's sale in the admin console (`admin.html`), writing `brand_sale_cycles` + `brand_sale_events` (`active_cycle_id` / `last_verified_status`). The old automated `scraper.js` was **removed** (Jun 2026); the now-frozen `brand_sale_events.sale_status` / `date_first_detected` / `scraper_error` columns remain in the DB but are unread. Admins find sales via the "open shop" link + crowd user-reports (the review nudge), not a scraper.

## Centre catalogue (Jul 2026 audit — main centres only)
The app tracks **24 UK centres** (owner's rule: every centre must be a main destination centre with lots of the tracked stores). The July 2026 audit (DECISIONS.md D17) removed six secondary centres (The Lexicon, Friars Walk, Queensgate, Broadmead, Touchwood, Bentall Centre — deactivated via `centres.active = false`, not deleted) and rebuilt the broken presence rows of eleven major centres (Lakeside had 0 tracked stores, Bluewater 5). Evans (B007) and Coast (B036) were dropped from the tracked brand set — online-only brands with no physical stores. Every live centre now has 11–56 tracked stores.

Centre identity: the UI uses **position codes** (`'C' + (index+1)` into `CENTRE_NAMES`); the DB keys centres by slug id, translated at the persistence boundary by name-matching (`ensureCentreIdMaps`). When adding/removing a centre, keep ALL centre-indexed structures in `index.html` in sync — `CENTRE_NAMES`, `CENTRE_DOMAINS`, `CENTRE_LOCATIONS`, `CENTRE_COORDS`, `CENTRE_HOURS`, every `PRESENCE` row (one column per centre, `test/presence.test.mjs` asserts 24), and the hardcoded `#centre-select` option list — plus a `centres` DB migration. The one-off audit scripts in `scripts/` (fetch-centre-directories etc.) still assume the pre-audit 30-centre index space; re-map their indices before reusing them.

## Design system (Jun 2026 brand convergence)
The app had drifted into two visual languages; these are now unified — don't revert:
- **One green per surface.** `--leaf` is the single action/brand green on **light** surfaces; `--tide-neon` (`#5EFFB0`) is the signal green on **dark** surfaces. The former `--green-bright` / `--green-deep` / `--green-light` are now **aliases onto leaf** (kept only so old call sites resolve) — don't reintroduce a third light-surface green. The chart's "whole centre" comparison line is a muted **sage-green** `#5E9E7E` (never blue — blue was the only blue in the app).
- **Two typefaces only: Playfair Display + Inter.** **Cormorant Garamond was retired** (a second display serif doing Playfair's job + a whole extra font download). Playfair = display/emotional headlines **and the logo wordmark**; Inter = everything functional incl. data figures (tabular-nums). The font `<link>` loads Playfair 400/600/700 (+italic) and Inter 400–700 — don't re-add Cormorant.
- **One calm cream canvas.** The generic mall line-art photo (`bg-mall.jpg`) was **deleted** — it added scroll-jank (two `fixed cover` layers) and a marooned-column look on desktop. Landing + centre view now share `--cream` with a faint top-light gradient; the dark tide vessel is the jewel against it. There's a desktop (`min-width:768`) block giving the landing real presence.
- **Graceful states.** `#main-content` is JS-rendered; a branded skeleton (`.tide-skeleton`) holds the space on load and a watchdog swaps in a friendly retry (`.tide-loaderror`) if the script/data never renders — never a blank middle.
- **Motion tokens** `--ease-out` / `--dur-1` / `--dur-2` exist for shared timing; route new transitions through them.

## Auth & registration (updated May 2026)

### What exists
The auth modal is a **4-step state machine** driven by `authShowStep(step)`:

| Step | ID | Purpose |
|------|----|---------|
| `email` | `#auth-step-email` | User enters email, clicks Continue or requests magic link |
| `signin` | `#auth-step-signin` | Returning user — password field, magic link fallback, "new here?" escape |
| `signup` | `#auth-step-signup` | New user — create password (8+ chars), "already have account?" escape |
| `magic` | `#auth-step-magic` | Confirmation: "check your inbox" shown after OTP send or email-confirm signup |

Key functions: `authContinue()`, `authSignIn()`, `authSignUp()`, `authMagicLink()`, `authShowStep(step)`.

### "My Tide" account panel
Signed-in users clicking the nav button open `#account-panel` (not the prefs wizard). It shows:
- Their email
- Saved centres list
- Peak alert + daily digest toggles (inline quick-save via `quickSavePref(key, value)`)
- "Edit shopping preferences" → opens prefs wizard
- Sign out

`onAuthBtnClick()` routes to `openAccountPanel()` if signed in, `openAuthModal()` if not.

### First-time onboarding
`onAuthStateChange` fires `openPrefsModal()` only when `!userPrefs` (no row in `user_preferences` table). Previously used a fragile gender-field check — don't revert to that.

### Preferences wizard (`#prefs-modal`)
4-step wizard: gender → style clusters → notifications + saved centres → preview. Opens automatically for new users and via "Edit shopping preferences" in the account panel. Saves via upsert on `user_preferences` with `onConflict: 'user_id'`.

## Verdict vocabulary (May 2026 — trend-only headlines)

Headlines on the centre card are **pure trend signals**. Recommendation language (`go`, `worth it`, `worth a visit`, `don't wait`, `skip`) is reserved for the **PEAK badge** — the only state where the dashboard tells the user to act. Every other state describes direction, not prescription.

**Score = % of tracked brands on sale.** `tide_score = round(brandsOnSale / totalBrands × 100)`. A user looking at the card can verify the score directly against "X of Y brands on sale". No freshness weighting, no anchor multipliers — the number on the gauge is the brand-density fact the card already shows. The old freshness-weighted formula made the headline drift on its own as brands aged, which is why a centre adding 10 new sales today could read OVER.

| Stage (internal) | Score | Verdict (stored in `centre_seer_scores.verdict`) | Headline word | Badge |
|---|---|---|---|---|
| Turning (cycle hasn't started) | 0 | `Quiet` | **QUIET** | — |
| Turning (early — small handful of brands) | >0, <15 | `Quiet` | **QUIET** | — |
| Rising | 15–<40 | `Rising` | **RISING** | — |
| High Tide (global) | ≥40 entering; holds 30+ **only while trajectory is RISING, or FLAT within `CREST_DROP_BAND` of the recent 14-day crest** — a confirmed FALLING (any score) OR a FLAT reading ≥4pts below that crest (D19) exits to Easing; re-entry from descent needs ≥40 + RISING | `Peak` | **PEAK** | **GO NOW** |
| High Tide (local peak) | ≥15, day trajectory flips RISING → FALLING | `Peak` | **PEAK** | **GO NOW** |
| Falling | 8–<30 post-peak | `Easing` | **EASING** | — |
| Low | <8 post-peak | `Over` | **OVER** | — |

**Quiet covers both score = 0 and 0 < score < 15.** A centre with one or two brands on sale reads the same as a centre with none — both are "nothing meaningful yet". The internal `stage` value is still `'Turning'` for back-compat with all the call sites that switch on stage, but every user-facing surface (vessel headline, count line, history chart) shows **QUIET**.

**Local peak:** every centre has a peak sale day, even ones that never break the 40% HIGH_TIDE_ENTER. `score.js` detects this as a one-shot trajectory flip (RISING → FLAT/FALLING) inside the climb path and emits `verdict='Peak'` for that single day; the front-end shows PEAK + GO NOW and the peak-alert email fires. The next day, `STAGE_FROM_VERDICT['Peak']='High Tide'` routes the centre through the descent branch and we transition to Easing automatically. Genuine ≥40 cycles hold PEAK while the trajectory is RISING or FLAT (the crest — a plateau at the top still reads PEAK) **until the decline is confirmed** (trajectory FALLING → Easing at any score, Jul 2026 D18 fix — the old score-only 40/30 hold kept every centre at "Go now" for weeks down the far side of the first recorded full cycle) or the score drops below 30. **Crest-distance release (Jul 2026 D19):** a slow decline off the crest — or a **carry-forward-frozen** score (no admin action → yesterday's score re-fed to `getTrajectory`, giving `diff≈0` → FLAT forever) — reads FLAT and never confirms FALLING, so the D18 rule alone left centres like WestQuay pinned at "Go now" / firing the "tide is in" peak-alert email while visibly declining (54%→49%). D19 adds a fourth exit: a **FLAT trajectory whose score sits ≥`CREST_DROP_BAND` (4pts) below the centre's recent 14-day crest** (`CREST_WINDOW_DAYS`) also releases the hold → Easing. It's FLAT-only (a RISING day near the crest still holds) and crest-distance-based, so it fires even when the score is frozen (compares to a PRIOR crest, not to movement) and doesn't age out; a plateau AT the crest (distance <4) still reads PEAK. The crest is the max of the already-loaded prior-60-day rows windowed to 14 days (`crestMap` in `score.js`, reusing `sixtyDayScoresRes` — no new query; `crestScores` buffer in `lib/tide-machine.js`, cleared on a broken trajectory window). Fixing the verdict → Easing stops the false peak-alert email automatically (`notify-high-tide` gates on `verdict=='Peak'`) — no email/copy change. Re-entry from descent requires ≥40 AND a sustained RISING trajectory, so an Easing bounce can't flap back to PEAK or re-fire the peak-alert email. This rule lives in BOTH `score.js` `getTideStage` and `lib/tide-machine.js` `stageStep` — change them together or the parity suite fails. `scripts/analyze-tide-cycles.mjs` (run via the manual `analyze-tide-cycles.yml` workflow) replays the stored score history through the rules to sanity-check verdict behaviour against real cycles. **Sustained-decline decay (Jul 2026 D20):** D19's release is FLAT-only, but a **steady slow slide of ~0.5–0.7 pts/day** never reached FLAT — its per-step drop was too small for the FLAT band (`TRAJECTORY_FLAT_BAND` 1.5) yet its 3-day span too wide for the stall check (`TRAJECTORY_STALL_RANGE` 1.5), so `getTrajectory` reported a **sticky RISING all the way down** and the FLAT-only crest release could never fire. That was the recurrence of the stale "Go now" (WestQuay / Festival Place / Westfield Stratford showing PEAK while every metric read "down from last week"). The root cause was the trajectory label, not the stage rule, so **the fix is in `getTrajectory`/`trajectoryStep`, not the crest release**: inside the sticky-RISING branch, **two consecutive down days at a fresh low** (`todayScore < recentScores[0] && recentScores[0] < recentScores[1]`) decay RISING → FLAT, which then lets the unchanged D19 crest release ease the centre once it's ≥4pts below its crest. A **one-day dip after a fresh high** (only the newest day down) still holds RISING — the deliberate one-day grace that stops a single noise blip dropping PEAK — and a genuine climb (today a new high) is never decayed. RISING-only and stateless, so `score.js` `getTrajectory` and `lib/tide-machine.js` `trajectoryStep` stay in parity — change them together too.

Legacy verdict strings (`Go now`, `Worth watching`, `Last chance — tide going out`, `Starting to build`, `It's over`, `Nothing on`) still resolve in every consumer (`score.js`, `index.html`, `summarise.js`, `notify-high-tide/index.ts`) so pre-rename rows render correctly. The next daily run rewrites the column with the new vocabulary — no migration needed.

**Alignment rules every consumer must obey:**
- The literal day-on-day brand-count change is brand-count-driven, so stage and delta cannot disagree. It is **not** rendered anywhere any more. ## "Newest Sales" panel — start date vs. discount change (Jul 2026)
The "Newest Sales" card (`renderCentreNarrativeCard` in `index.html`) surfaces a brand when EITHER its sale started in the last `FRESH_WINDOW_DAYS` (**7** — "this week", deliberately equal to the hero depth row's `DEEPEN_WEEK_DAYS` so the card and the hero count the same deepened brands; was 5 pre-Jul 2026) days OR its discount % last changed in that window — not start date alone. This matters because an admin routinely revisits an old, already-listed cycle (e.g. bumping White Stuff from 50% to 60% on day 18) via Edit/Increase in `admin.html`; without the second condition that update would never surface here, which read as a bug ("I updated the %, why isn't it showing?").

- `brand_sale_cycles.pct_changed_date` (migration `20260707_add_pct_changed_date.sql`) tracks the date `max_discount_pct` was last actually changed, backfilled to `start_date` for existing rows. `admin.html` only bumps it to today when the new % differs from the previous value — a routine re-confirm with the same % (crowd-report quick action, `confirm_start` reusing an open cycle) must NOT make an old sale look freshly updated. `applyIncrease` always bumps it (that action's whole purpose is raising the %); `applyEdit`/`confirm_start` compare against the prior % first.
- `index.html` derives `PCT_CHANGE_DAYS`/`PCT_CHANGE_DATE`/`SALE_START_DATE` per brand alongside the existing `DAYS_RUNNING`, and `calcCentreScore` carries `pctChangeDays`/`startDate`/`pctChangeDate` onto each on-sale brand object.
- **Two tinted sections (Jul 2026 redesign, owner-directed — prototypes in `newest-sales-prototypes.html`; the "tinted sections" band structure + the "delta chips" row anatomy shipped).** The list is split into two colour-keyed bands, each a full-width tinted block with its count (`.rank-sec`): a leaf-tinted **"New sales"** band (this week's fresh starts, newest-first by start date) and an amber-tinted **"Just got deeper"** band (already-running sales whose % was raised this week — `isDeepened`, delegating to the shared `isDeepenedBrand(b, FRESH_WINDOW_DAYS)` — newest-first by raise date; wording mirrors admin.html's Insights tab). Ties break to the deeper discount; an empty section is omitted entirely. Deepened rows carry a 2px amber left edge so membership survives scrolling past the band, and swap the plain "Up to N%" figure for an amber **▲ delta chip** — "▲ 50 → **70%**" (`.rank-chip`), "▲ now 70%" when there's no usable prior %, or a bare "▲ updated" beside the label for a non-percent sale type. The chip is the sole home of the was→now fact. A quiet muted age line ("4 days" / "today", `.rank-age`) sits under the value on every row; there are NO per-row rank numbers, day pills, or NEW tags (the leaf band carries "new" for every row inside it — an interim single-merged-list design used per-row NEW tags, and the owner found the two kinds of freshness too hard to tell apart). Earlier failed presentations, in sequence: per-row UPDATED badge + "Started {date} · % updated {relative}" meta sentence; two floating caps labels over identical-looking rows with per-group rank numbers + a "Was X%, now up to Y% · started {date}" sentence (didn't register); one merged recency list with NEW tags (lost the split). Amber = "deepened" is unchanged; the ▲ chevron matches the centre hero's weekly movement vocabulary (`tideWeekFromHTML`).

## Admin "Sale history" tab — cycle merge (Jul 2026)
Third tab in `admin.html`, deliberately styled as the shopper-facing shop detail sheet (`th-*` classes mirror index.html's `ts-*` palette/anatomy: rhythm chart + "Every sale on record"). Purpose: when an operator mistakenly ends a sale one day and re-confirms it as a "new" one the next, two `brand_sale_cycles` rows describe one real sale — the rhythm chart dips-and-recovers and dates read wrong. The admin picks a shop, **selects two back-to-back entries** (tap-to-select, max 2, oldest pick evicted), and a gold merge bar previews the single sale they become; same-% pairs are the intended case, differing % shows a warning (merged sale keeps the deeper %). Pure field logic lives in `lib/merge-cycles.js` (`computeMergedCycle` — earlier row survives with its id/start_date so the one-shot brand-sale alert can't re-arm; % = max of both; `pct_changed_date`/`prior_discount_pct` only rewritten when the later cycle was genuinely deeper, so a same-% merge never fakes a deepening; unit-tested in `test/merge-cycles.test.mjs`). DB sequencing in `mergeCycles()` is order-critical: repoint `brand_sale_events.active_cycle_id` → survivor, **then** delete the later row, **then** re-open the survivor — so the FK's ON DELETE SET NULL never fires and the one-open-cycle-per-brand unique index never sees two open cycles. Logs a dedicated `merged` verb (migration `20260718_add_merged_action.sql` extends the `admin_review_log` action CHECK) with a before/after date-range note, then fires the instant rescore. The log write is deliberately non-fatal: on a DB where the migration hasn't run, the CHECK rejects `merged` and the code falls back to `edited` with a "Merged: …" note (and even a total log failure only warns) — the cycle writes have already committed by then, so surfacing "Merge failed" would be false. **Stored-history heal**: the gap a merge closes still carries a fake dip in `centre_seer_scores` (the brand was counted off-sale on those days), so the post-merge rescore POST includes `heal_gap: {brand_id, from_date, to_date}` (`from` = earlier cycle's original `end_date` inclusive — the "Sale ended" intraday rescore stored that whole day as off; `to` = later cycle's `start_date` exclusive). `/api/rescore` validates the payload and calls `score.js`'s `healMergedGap` (service key) BEFORE `runScoring`, which +1s `brands_on_sale` (clamped) and re-derives `tide_score` on every stored row in the gap across the brand's centres — the subsequent `tide_history` rebuild then picks the healed rows up in the same call. Past days' `verdict`/`trajectory`/`avg_discount_pct` are deliberately untouched (not rendered historically / not honestly recomputable). Pure helpers `healGapDates`/`healedScoreRow` live in `lib/merge-cycles.js`; a heal failure downgrades the rescore response to 502 (scores refreshed, dip may remain) rather than aborting.

## Notification emails (May 2026)
Supabase Edge function `notify-high-tide` runs three passes, gated by the POST body so one function serves two schedules. Scheduled in-repo by `.github/workflows/notify.yml` (NOT pg_cron — unschedule any old `notify-high-tide-daily` job to avoid a daily digest).

| Pass | Trigger | Schedule (body) | Gating column |
|---|---|---|---|
| Peak alert | a saved centre hits **Peak** (verdict `Peak`) | daily 11:00 UTC, `{}` | `email_alerts` |
| Brand-sale alert | a followed brand's sale cycle **starts today** OR its discount **deepens today** | daily 11:00 UTC, `{}` | `brand_sale_alerts` (+ respects `excluded_brand_ids`) |
| Weekend digest | ≥1 saved centre at Rising or above | **Friday 19:00 UTC**, `{"digestOnly":true}` | `daily_digest` |

**One email per address per run — never per sale (Jul 2026).** The function also has a pass 4 delivering `seo_alert_signups` (the accountless opt-ins from static SEO pages + the in-app one-field forms). As first shipped it sent **one email per signup row**, so an address opted into N brands got N separate emails the day a national event started them all — per-sale spam. The rule now: **each pass sends at most ONE email per address per run.** Pass 2b already grouped per user (`renderBrandSaleDigestEmail`); pass 4 now groups all of an address's firing rows the same way (multi-brand → the digest render; same brand at two centres → one item with `centre2`; brand news wins over a same-day peak hit), pass 1 caps at one peak alert per address (centres sorted by score, strongest wins — `peakAlertedAddrs`), and cross-pass dedupe (`peakEmailed` / `brandEmailed`) stops pass 4 repeating what an account email already carried today. A consolidated email spanning multiple signup rows links `/api/unsubscribe?token=…&all=1`, which clears **every** signup for that address (the footer promises as much). Do not reintroduce per-row sends.

UI toggles in the account panel map to these three columns. Don't rename the columns — only the human-readable copy. The digest is **weekly (Friday)**, not daily — the email copy hard-codes "Friday" and that is now correct. The daily run is **11:00 UTC** (an earlier version of this note said 07:00 — notify.yml deliberately fires ~1h after the 10:00 UTC scorer so today's `centre_seer_scores` rows exist). Brand-sale "started today" = active cycle `start_date == today` or `date_first_detected == today`; true for one day only, so it self-dedupes (no sent-state table). `digestOnly` runs pass 3 only; the default daily call runs passes 1+2 and skips the digest.

**Deepened-brand alert + digest depth (Jul 2026).** The brand-sale pass also fires when a followed brand's live sale **cuts deeper** — `deepenedToday`: active cycle `pct_changed_date == today && != start_date` (backfilled to `start_date` on new cycles, and `admin.html` only bumps it on a genuine % change, so it's one-shot like `startedToday`; the two are mutually exclusive by construction, started wins, so a same-day open+Increase sends one started email with the raised %). Same gate (`brand_sale_alerts` + `excluded_brand_ids`); mixed same-day news groups into ONE summary email per user (`renderBrandSaleDigestEmail` is kind-aware, new sales listed first). The single-brand deepened email (`renderBrandDeepenedEmail`) keeps the **LEAF** Brand-Sale hero/CTA with **AMBER only as inline accent** on the "was 50%, now up to 70% off" numbers (`prior_discount_pct`; clause dropped when unrecorded) — an all-amber hero would masquerade as the Peak alert. The Friday digest cards gain a depth line — "Average discount 45% · 3 cut deeper this week" (`centre_seer_scores.avg_discount_pct` + `deepenedWithin` over `DEEPEN_WEEK_DAYS` = 7, mirroring the hero row; clause omitted at zero, whole line omitted when avg is null — same `+null === 0` guard) — and `digestVerdictFor`'s **falling** line gains "…but remaining sales are cutting deeper" when any brand there deepened this week. All copy stays trend-only; "Go now" remains exclusive to the high/peak bucket. Response JSON gains `brandsDeepenedToday`; brand log entries carry `kinds`.

## Instant rescore (May 2026)

`/api/rescore.js` is a Vercel serverless function that wraps `runScoring()` from `score.js`. The admin panel (`admin.html`) fires a fire-and-forget POST to it after every successful `applyAction` / `applyEdit`, so the public site reflects admin edits within a couple of seconds without waiting for the 10:00 UTC cron. Carry-forward in `score.js` (if no brand at the centre has `last_verified_date == today`, reuse yesterday's brand counts) keeps the rescore cheap and lets the state machine re-derive (e.g. a Peak rolls to Easing) without forcing the admin to re-verify a brand they already know is on sale, and preserves narratives on untouched centres.

**The rescore is fire-and-forget, but failures are surfaced** — `triggerRescore` raises a persistent error banner (`showRescoreWarn`/`clearRescoreWarn` in `admin.html`) when the POST returns non-OK or can't be reached, tailored to the cause (401/403 = expired admin session; 502 = scores updated but `tide_history` stale; other = full lag until the daily cron). The banner auto-clears on the next successful rescore. Previously a failed rescore left the admin with a success toast while the public store silently went stale. The safety net is the daily cron: `node score.js` (the CLI path) now calls `runScoring({ forceFresh: true })`, which **bypasses carry-forward and recomputes every centre from live brand state**. This self-heals any drift within 24h — a sale confirmed *after* the previous cron, or a silently-failed intraday rescore, would otherwise stay frozen forever, because the carry-forward gate keys off `last_verified_date == today` while "on sale" is also true via an `active_cycle_id` opened on a prior day. A fresh compute reads the same persistent on-sale state, so it matches carry-forward when nothing changed and corrects it when it has. `forceFresh` nulls narratives, but the summariser runs immediately after the scorer in the same workflow and repopulates them. The intraday rescore deliberately leaves `forceFresh` off. The **front-end also defends against this independently**: the 60-day chart's "today" point is anchored to the live headline value (`pctOnSale`), so the curve can never sit under QUIET while the headline reads PEAK even if the stored `tide_history` is momentarily stale (`buildTide60Series` takes `pctOnSale`).

**Required Vercel env vars** (Project Settings → Environment Variables — mirror the GitHub Actions secrets):
- `SUPABASE_URL`
- `SUPABASE_SERVICE_KEY`

`score.js` is now safe to import as a module — env-var checks and the supabase client are lazy, and the CLI entry is guarded by an `import.meta.url === file://…argv[1]` check so daily-cron behaviour is unchanged.
