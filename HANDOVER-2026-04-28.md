## What was done this session

### 1. score.js — full rewrite to Tide Score Spec v2.0
`score.js` was completely rewritten to implement the Tide Score Specification v2.0:

- **Freshness curve** (replaces Apple Model ripeness):
  - Days 1–7: 1.0 (full freshness)
  - Days 8–21: linear decay from 1.0 → 0.5
  - Days 22–42 (DECAY_MAX): linear decay from 0.5 → 0.1
  - Beyond DECAY_MAX: excluded entirely (freshness = 0)

- **Anchor brand multiplier**: `ANCHOR_MULTIPLIER = 1.5` applied to six anchor brands: Next (B001), M&S (B002), Zara (B003), H&M (B005), Uniqlo (B013), River Island (B023)

- **Six tide stages** replacing old Apple Model stages:
  | Stage | Score range | Trajectory |
  |-------|-------------|------------|
  | Flat | 0 | any |
  | Turning | 1–24 | rising/flat |
  | Rising | 25–49 | rising/flat |
  | High Tide | 50–100 | rising/flat |
  | Falling | 25–100 | falling |
  | Low | 1–24 | falling |

- **Trajectory** via 3-day rolling average: ±1.5 band = Flat, above = RISING, below = FALLING. New centres default to RISING.

- **Writes to `centre_seer_scores`**: `tide_score`, `phase`, `verdict`, `bluf`, `trajectory`, `brands_on_sale`, `total_brands`, `top_brands`, `avg_discount_pct`

- **Writes `tide_history`** to `centres` table: up to 60 days of `{date, score}` JSON, ready for sparkline display

### 2. index.html — dashboard updates

#### Scoring logic
`index.html` now re-implements the v2.0 scoring algorithm client-side (architectural debt: ideally reads from `centre_seer_scores`). Functions updated:
- `brandFreshnessScore(daysRunning)` — piecewise linear per spec
- `calcCentreScore(centreIdx)` — uses anchor multiplier + freshness, excludes brands beyond DECAY_MAX
- `getTideStage(score, trajectory)` — six-stage logic with trajectory
- `computeTrajectory(score, history)` — 3-day rolling average from history array
- `stageStyle(stage)` — colour + label + message for all six stages

#### History chart
- Smooth Catmull-Rom bezier with triangular weighted moving average (window ±5) for sub-pixel precision
- 60-day sparkline with gradient fill, hollow "today" dot, trajectory label
- **Mock data shown when no live Sheets data** (badge: "Preview — no live data yet")
- X-axis: "Nd ago / Nd ago / Today"

#### Arc gauge widget (NEW — this session's main feature)
Replaces the old linear ripeness bar. Rendered by `renderArcWidget(stage, color)`:
- SVG semi-arc: `M 25,95 A 190 190 0 0 1 335,95` (center (180,205), r=190, peaks at (180,15))
- Gradient track: grey → green → lime → amber → orange → dark red
- Six stage labels (FLAT / TURNING / RISING / GO NOW ★ / FALLING / LOW), active label highlighted in phase colour
- Hollow dot (white fill, coloured stroke r=7.5) positioned at the current stage

#### Empty-state text
Updated "Is the moment ripe?" / 🍎 → "When does the tide come in?" / 🌊

## Known issues / things to verify

### Arc widget — confirm it renders
The arc widget was failing to display due to wrong SVG sweep direction (sweep=0 curved downward, outside viewport). Fixed in commit `20a8f9a` to sweep=1 with corrected geometry. 

**To test**: download `index.html` from the branch, open it, select a centre. The arc should appear between the score number and the message text, arching upward like a rainbow with a coloured dot at the active stage. If it still doesn't show, open browser DevTools → Console and check for JS errors.

### History chart trajectory label
The chart still shows "→ Holding" in some edge cases. Should say "→ Flat". Check `renderHistoryChart` — the trajectory label uses the string returned by `computeTrajectory` which returns `'FLAT'`, so the display string should read `→ Flat`. If "Holding" appears, there's a stale cached version.

### `index.html` computes scores client-side
Spec §9.1 says scores should come from the database. The dashboard re-computes from the Sheets CSV because Google Sheets provides a free public CORS-friendly endpoint. Supabase would require exposing an anon key. This is acknowledged technical debt — fine for now.

### `score.js` decay curve final segment
The spec v2.0 says days 22–DECAY_MAX should decay to **0.1**, but the current implementation decays to **0.1** (`0.5 - (days-21) * (0.4 / 21)`). Double-check against the spec when you have the final numbers. `DECAY_MAX = 42` and the multiplier `0.4/(42-21)` ≈ 0.019/day.
